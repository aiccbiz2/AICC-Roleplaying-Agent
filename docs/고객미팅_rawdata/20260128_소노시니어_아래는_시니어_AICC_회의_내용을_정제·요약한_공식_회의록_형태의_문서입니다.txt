1. 현재 인프라 및 구성 현황 확인
주요 사실
•	SBC 없음, 교환기 및 PBX는 대부분 Cisco 기반.
•	CTI는 Cisco 가상화 환경 존재하나 주 솔루션은 타사.
•	PDS 없음
•	녹취는 별도 업체 솔루션 사용 중, 화자분리/실시간 스트리밍 미지원.
o	업그레이드 시 화자분리·STT 연동 가능.
•	온프레미스 구축 선호(개인정보/보안 정책 이슈)
o	STT, TTS, Voice Gateway, 녹취 등 민감 데이터 처리 요소는 반드시 내부 구축.

3. DB 및 데이터 흐름 구조
DB 구성 방향
•	현재 문서상의 “DB”는 개념적 범위이며 실제는 세분화 필요:
1.	고객 식별 DB (PII 분리) – 기관계 연동 전에 키 기반 식별
2.	기관계 연동 DB – 예약 내역, 이용 내역, VOC 등
3.	여행 요소 DB(별도 개발) – 여행사 입력 UI 포함
4.	고객 성향 DB – 상담/이용 데이터 기반 자동/수기 입력
•	개인정보 최소화 위해 식별키 기반 조회 구조 희망 – 소노 기간계별로 통일된 키가 없음
(개인정보는 상담사 화면에서만 표출)
여행 요소 DB 입력 UI
•	외부 여행사 계정으로 별도 관리
(SSO 아님 → 보안 위험 지적됨 → 내부 재검토 예정)
________________________________________
4. 맞춤형 여행 추천 + 콜봇 구조
1단계–2단계 범위 구분
•	현재 문서의 구성도는 “최종 완성형(본사업 기준)”
•	시범사업(1단계)에서는 일부 축소 가능하나 기본 인프라는 1단계부터 구축 선호.
주요 Use Case 확인
고객 상담 후 → 일정표 생성 → 예약 필요 항목 자동 콜 발신
•	예약이 필요한 식당/서비스는 여행 DB에서 “예약필수” 여부로 판단.
•	예약봇이 식당에 다음 내용을 전달:
o	방문 날짜/시간
o	인원수
o	특이사항(예: 저염식 요청 등)
o	예약 가능 여부(예/아니오)
•	실패 시 재발신 또는 상담사 처리.
→ 해당 기능은 1단계 시범사업 범위에 포함.

5. SaaS vs On-Premise 논의
SaaS 활용 제안 (LG U+ 측)
•	시범사업 예산 절감 및 기능 검증 위해 일부 SaaS 활용 제안:
o	Outbound simple 콜봇
o	일정 추천 엔진 등 비민감 기능
고객사 정책
•	개인정보 포함 기능은 SaaS 절대 불가
•	STT, TTS, 콜봇, CRM 등은 반드시 On-Premise
•	단, SaaS 제안은 가능: RFP 평가 시 참고
________________________________________
6. 상담사 애플리케이션(AP) 및 상담 어시스트
요구사항
•	기존에 명확히 정의되지 않아 RFP에 세부적으로 추가 중.
•	주요 기능:
o	고객 정보 연동
o	상담 이력 마이그레이션(2025.1.1 이후)
o	녹취 STT 기반 학습 데이터 생성 (과거 5~10년 치도 검토 필요)
o	일정 추천 결과 상담 화면 연동
o	상담 중 도움 요청/코칭(삼자 통화 또는 팝업 기능)
o	알림톡/SMS/PDF 전송 기능 연동
