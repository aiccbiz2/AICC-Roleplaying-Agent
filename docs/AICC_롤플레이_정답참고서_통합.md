### A-1. 보험사 AICC 도입 초기 미팅

| **데이터 소스** | 회의록 직접 인용 (한화손해보험 AICC도입 미팅 060323, 67분) |

**Q1. 금융권 구축 레퍼런스가 있나요?**

| 핵심 포인트 | 정답 |
|------------|------|
| KB금융그룹 FCC | 7개 계열사(KB손해보험, KB생명 등)에 자체 STT 도입 중. 기존 엔솔루 STT를 LG유플러스 자체 STT로 대체 진행 |
| KB손해보험 PoC | RX/TX 각 100건(총 200건)으로 PoC 수행. 180건 학습, 20건 검증 방식 |

| LLM 콜봇 사례 | **"금융권에 LLM 기반 콜봇 상용 사례는 아직 없다"고 솔직 인정**. 유플러스 자체 5,000석 콜센터도 LLM 콜봇 미도입 (NLP/시나리오 기반 운영 중) |

**Q2. STT 인식률은 어느 정도인가? (금융 용어 학습)**

| 핵심 포인트 | 정답 |
|------------|------|
| 기본 모델(텔코) | 80~90% (금융 도메인 학습 전) |
| 금융 특화 학습 후 | **약 95%** (KB손해보험 PoC 기준: 상담원 98%, 고객 92%) |
| 금융 모델 범용성 | KB 종속 아닌 **금융권 전반에 범용 사용 가능** |
| 학습 도구 | 웹 형태 사전 관리 + 파일 업로드 기능 제공. 금융 용어/상품명 파일 업로드 후 클릭 적용 가능 |
| 인식률 측정 | 파일 단위 인식률 측정 기능 웹 제공 (정답지 대비 비교). 대량 데이터는 서버 레벨 처리 |
| 표준 인식률 범위 | 일반적으로 **93~96%** 수준 |

**Q3. 레거시 인프라 변경 없이 STT 도입이 가능한가?**

| 핵심 포인트 | 정답 |
|------------|------|
| 인프라 변경 | **인프라 구성의 변경은 불필요**. 게이트웨이 정도만 추가 |
| 하드웨어 | 일부 하드웨어 스펙 추가 가능성 있으나 대규모 변경 없음 |
| 녹취 연동 | 기존 동방녹취에서 RX/TX 분리된 음원 파일로 STT 연동 가능 |
| 실시간 미러링 | 녹취 쪽에서 실시간 음원 미러링 인터페이스 필요 가능 (녹취 업체와 협의) |
| 구독형 제공 | 구축형 외에 **구독형으로도 제공 가능** |
| 구축 일정 | STT/TTS만 적용 시 최소 **4개월**, 음성봇 전체 구축 시 **6~8개월** (업무 범위에 따라 최대 1년) |

**Q4. 센터별(인바운드/TM) 각각 구축해야 하나?**

| 핵심 포인트 | 정답 |
|------------|------|
| 큰 시스템 | 비슷하게 구성 가능 |
| 세부 차이 | 인바운드(고객센터): 지식 추천/고객 의도 파악 / TM(아웃바운드): 추천 상품, 대본, 후처리가 다름 |
| STT 공유 | 동일 STT 엔진 사용 가능 |
| 사이트 라이센스 | CS, TM, 보상 등 여러 업무에서 사용 가능 (단, 유저 수는 정해야 함) |
| 모바일 녹취 | 유선 녹취는 바로 가능. 모바일 녹취 STT는 별도 검토 필요 |

**Q5. KMS 전담자가 필요한가?**

| 핵심 포인트 | 정답 |
|------------|------|
| 초기 세팅 | KMS 데이터를 LLM이 읽을 수 있게 정제하는 과정은 기술 조직에서 수행 |
| 운영 단계 | **일 단위 배치**로 자동 업데이트. 현업 부서가 수시 업데이트하면 센터에서 가져가는 구조 |
| 유플러스 사례 | 초기 **8명** 투입 -> AI 인더루프 자동화 후 **전담자 불필요**. 수작업 5,760분 -> 40분 단축 |
| 신규 시나리오 | 현업에서 직접 추가 어려움 -> IT와 짝으로 투입 권장. 계약 시 **연간 무상 추가 건수** 합의 필요 |

**함정 대응 가이드**
- **금융권 LLM 콜봇 사례 없음**: "솔직히 금융권에서 LLM 콜봇 상용화 사례는 아직 없습니다. 유플러스 자체 콜센터도 NLP 기반이고, LLM 콜봇은 5월 오픈 예정입니다. LLM은 물어볼 때마다 다른 답을 하는 이슈가 있어 금융권은 시나리오 기반이 현실적입니다." -- 솔직한 인정이 신뢰 구축의 핵심
- **녹취 데이터 확보 방안**: "내부 데이터 외부 반출에 보안 이슈가 있다면, KB손보 사례처럼 RX/TX 각 100건의 음원 파일을 제공받아 PoC 진행 가능. 데이터 증강 기술로 소량 데이터에서도 유의미한 결과 도출합니다."

### A-2. 증권사 콜봇 도입 탐색

| 항목 | 내용 |
|------|------|
| **출처** | 한국투자증권 |
| **데이터 소스** | 패턴DB 활용 + 대신증권 회의록 참조 (동일 금융/증권 도메인) |

**Q1. 콜봇 완전 무인화가 가능한가?**

| 핵심 포인트 | 정답 |
|------------|------|
| 결론 | **완전 무인화 불가**. AI를 싫어하는 고객, 돈/계좌 관련 크리티컬 문의는 반드시 상담사 연결 필요 |
| 현실적 목표 | 단순 반복 문의(시세, 영업시간, 잔고)를 콜봇이 흡수 -> 상담사 연결 콜 자체를 줄이는 것 |
| 대신증권 참조 | "보이스봇은 지금 도입해도 솔직히 기대하기 쉽지 않다." 주문까지 가능한 시점은 2~3년 소요 |
| 금융 리스크 | 오답변 시 금융 사고 직결 -- 금융권이 콜봇/LLM을 가장 꺼리는 이유 1순위 |
| 권장 접근 | 내부 직원용(어드바이저, KMS) 1년 운영 -> 검증 후 대고객 서비스 확대 |

**Q2. 콜봇 완결률은 보통 얼마나 되나?**

| 핵심 포인트 | 정답 |
|------------|------|
| 유플러스 사례 | 룰 베이스 콜봇 기준 연간 **800만 건** 고객 문제 해결 |
| LLM 챗봇 효과 | 대화 건수 약 **60% 감소**, NPS **50포인트** 상승 |
| 핵심 변수 | 업무 복잡도/도메인에 따라 상이. "콜봇이 처리할 업무"를 먼저 정의해야 완결률 논의 가능 |
| 룰 베이스 장점 | 정형화된 업무는 완결률 높음. 실수 없고, 할루시네이션 위험 없음 |

**Q3. 구독형/구축형 비용은?**

| 핵심 포인트 | 정답 |
|------------|------|
| 구독형 | 채널당 월 과금 방식. 소규모 시작 가능. BPO 10~15명 규모라면 초기 부담 적음 |
| 구축형 | 수억 단위 (업무 범위에 따라 변동). STT+콜봇+어드바이저 풀 구축 시 6~12개월 |
| 채널 구조 | STT 상담원 1명당 **2채널** 필요 (RX/TX 분리) |
| 전환 | 구독형 테스트 후 구축형 전환 가능. 아키텍처 차이 있으므로 일부 재구축 요소 발생 |

**Q4. KT 대비 기술 수준 차이는?**

| 핵심 포인트 | 정답 |
|------------|------|
| 원칙 | **경쟁사를 근거 없이 폄하하면 역효과**. 객관적 사실 위주 답변 |
| 차별점 | 자사 5,000석 콜센터에서 직접 운영하며 검증한 솔루션. 운영 노하우 + 솔루션 개발 동시 보유 |
| 원벤더 장점 | STT, 어드바이저, QA, 콜봇 모두 자체 개발 -> "서로 책임 떠넘기기" 문제 없음 |
| 권장 표현 | "저희 솔루션의 검증된 성과"를 수치로 제시하는 것이 효과적 |

**Q5. BPO 외주 10~15명 대체 가능한가?**

| 핵심 포인트 | 정답 |
|------------|------|
| 결론 | 완전 대체 불가. 단순 문의 콜봇 흡수 + 업무 효율화로 **인원 최적화** 가능 |
| 유플러스 사례 | 2년에 걸쳐 **507명 자연 감소** (인위적 해고 아닌 퇴사자 미충원) |
| 접근법 | 콜 데이터 분석 -> 콜봇 처리 가능 업무 식별 -> 단계적 적용 |
| 효과 시점 | 구축 후 바로 효과가 아닌, 데일리 피드백 루프 통한 지속 개선 필요 (최소 6개월~1년) |

**함정 대응 가이드**
- **경쟁사(KT) 비교**: "KT도 좋은 솔루션을 보유하고 있습니다. 다만 저희는 자체 콜센터에서 2년간 운영하며 검증한 피드백 루프와 운영 노하우가 차별점입니다." -- 근거 없는 우위 주장 금지
- **완전 무인화 기대치 조정**: "무인화보다 '스마트화'가 맞는 표현입니다. AI가 단순 업무를 흡수하고, 상담사는 복잡한 고가치 상담에 집중하는 구조입니다."

### A-3. 상담 어드바이저(KMS/RAG) 상품 소개

| 항목 | 내용 |
|------|------|
| **출처** | 우리금융캐피탈 / FIS |
| **데이터 소스** | 회의록 직접 인용 (우리금융캐피탈 다이퀘스트 상품소개 QnA 260311, 48분) |

**Q1. 챗봇+채팅 통합 솔루션이 있나?**

| 핵심 포인트 | 정답 |
|------------|------|
| 통합 제공 | 어드바이저(상담 지원) + 챗봇(대고객) + 에이전트 빌더를 **DELMA 플랫폼**에서 통합 제공 |
| 워크플로우 | 에이전트 빌더에서 질문 분류기, 검색, 응답 생성 등을 드래그&드롭으로 구성 가능 |
| 모델 선택 | 대화창에서 모델 선택하여 답변 비교 가능. 라마, QWen, DPT 등 다양한 모델 탑재 |
| 멀티턴 | 문맥 유지 대화 지원. 앞선 대화 맥락 기반 답변 제공 |

**Q2. 어드바이저는 별도 구매인가, 패키지인가?**

| 핵심 포인트 | 정답 |
|------------|------|
| 패키지 구성 | "디 플랫폼"이라는 이름 안에 어드바이저, 상품 추천(디필릭), ML옵스, LLM옵스를 **통합 제공** |
| 하드웨어 | 딥러닝 상품 추천 에이전트는 별도 GPU 서버, 플랫폼은 DELMA로 일괄 견적 |
| 분리 견적 | 요청 시 솔루션별 분리 견적 제공 가능 |
| STT 별도 | 기존 콜봇 도입 시 납품된 STT 라이선스 잔여분 활용, 연동 비용만 추가 |

**Q3. VDI 없이도 사용 가능한가?**

| 핵심 포인트 | 정답 |
|------------|------|
| VDI 불필요 | **웹 기반 솔루션이므로 VDI 없이 사용 가능** |
| 모바일 지원 | 모바일/태블릿에서 웹 기반 접근 가능. 외부 협력 직원 핸드폰 사용 가능 |
| 구현 방식 | 홈페이지 WAS에 컨테이너 추가 또는 별도 사이트 구성 |

**Q4. 기존 상담 AP(엑스프레임)와 연동 가능한가?**

| 핵심 포인트 | 정답 |
|------------|------|
| 연동 방식 | 엑스프레임 안에 iframe보다 **별도 창으로 띄우는 방식** 권장. API 형태 제공 가능 |
| 콜 인입 연동 | 소프트폰 전화 이벤트 -> URL 호출로 미니 어드바이저 자동 팝업 |
| 기관계 연동 | 고객번호/대출번호 인자값 전달 -> 기관계 데이터 자동 조회/노출 가능 |
| 부서별 차별화 | 대출 부서=대출 이력, 렌터카 부서=차량 정보 등 **부서별 다른 데이터 자동 세팅** |
| 후처리 연동 | 상담 코드 추천, 요약 등 후처리 데이터를 API로 기관계 DB에 적재 가능 |

**Q5. AI 도입 효과를 어떻게 측정하나?**

| 핵심 포인트 | 정답 |
|------------|------|
| 유플러스 수치 | 콜당 톡타임 **30초** 감소, 수작업 콜당 **19초** 감소, 어드바이저 도입 후 콜타임 **20%** 감소 |

| 고객별 맞춤 | 콜 인입 시 고객 정보 기반으로 납부 이력, 사용량, 계약 정보 등 자동 조회/표시 |
| 부서별 다른 정보 | 저장소 분리 + 기관계 연동으로 부서별 다른 데이터 자동 노출 |
| 사용자 화면 커스텀 | 사용자별 화면 구성 개인화 가능. 불필요한 기능 끄기 가능 |
| 우리캐피탈 참조 | "고객 발화에 따라 능동적으로 필요한 답변을 미리 찾아서 보여주는 형태" |

**Q2. 다국어 STT가 가능한가?**

| 핵심 포인트 | 정답 |
|------------|------|
| 한국어 특화 | 익시 STT는 **한국어 특화 모델**. 인식 정확도 **97%** (자사 기준) |
| 다국어 | 기본적으로 한국어 중심. 외국어 STT 필요 시 별도 모델 검토 필요 |
| 도메인 학습 | 도시가스 고유 용어(열량 단위, 배관 용어)는 사전 등록 및 도메인 특화 학습으로 인식률 개선 |
| 소음 필터링 | STT에 소음 필터링 기능 -> 외부 통화 시에도 정확도 유지 |

**Q3. LLM 토큰 비용은?**

| 핵심 포인트 | 정답 |
|------------|------|
| 온프레미스 권장 | 온프레미스(SLM) 기반이면 **토큰 과금 없음**. GPU 장비 초기 투자만 |
| SLM 성능 | QWen 32B, GPT OSS 120B 모델로 **GPT-4.0 수준 이상** 성능 (유플러스 검증) |
| 하이브리드 전략 | 단순 분류/요약은 경량 모델(BERT, 익시젠 2.4B), 복잡한 RAG는 대형 LLM -> **용도별 모델 분리**로 비용 최적화 |
| 모델 교체 | DELMA 플랫폼에서 모델 선택/교체 가능 -> 모델 종속 리스크 없음 |

**Q4. 보이는 콜봇(Visual IVR)이 가능한가?**

| 핵심 포인트 | 정답 |
|------------|------|
| 운영 중 | 유플러스가 **실제 운영 중**. "콜봇의 멀티모달 + 보이는 시나리오" |
| 구현 방식 | 음성 통화 중 고객에게 화면 동시 제공. URL 발송 -> 화면에서 정보 확인/입력 |
| 활용 사례 | 청구서 실시간 화면 제공, 이메일/주소/계좌번호 화면 입력, 가입 절차 안내 |
| 효과 | 음성만으로 전달 어려운 긴 내용 -> 텍스트+화면으로 전달도 향상, 재인입 감소 |
| 도시가스 적용 | 요금 조회, 검침 일정, 가스 안전 점검 예약 확인 등에 적용 가능 |

**Q5. AI 오인식 시 재학습은?**

| 핵심 포인트 | 정답 |
|------------|------|
| AI 인더루프 | TA 분류 코드와 상담사 실제 선택 코드 **불일치 시 자동 샘플링** |
| 자동 재학습 | LLM 정답 판정 -> 핵심 발화 추출 -> 벡터DB 임베딩까지 자동화. **5,760분->40분** 단축 |
| STT 자동 개선 | STT 데이터 자동 개선 기능 **7월** 적용 예정 |
| 오류 제보 | 리더/평가자가 포털에서 오류 신고 -> 자동 반영 |
| 데일리 피드백 루프 | STT, TTS, 요약, 분류 각 요소 목표 수준 세팅 -> 데일리 모니터링 |
| 학습 도구 | 웹 기반 사전 관리/학습 수행 기능. 파일 업로드 후 클릭 적용 + 인식률 즉시 확인 |

**함정 대응 가이드**
- **멀티 LLM 선택 기준**: "용도별로 최적 모델을 분리합니다. 단순 분류는 BERT, 지식 추천은 RAG+대형 LLM, 코드 추천은 별도 파인튜닝 모델. DELMA 플랫폼에서 모델을 쉽게 교체할 수 있어 모델 종속 리스크가 없습니다."
- **다국어 STT 한국어 특화**: "한국어 모델은 97% 인식률을 보장하고, 타 언어는 IVR에서 언어 선택 후 해당 언어 전용 모델로 분기합니다. 하나의 모델로 처리하면 한국어 인식률이 떨어지므로 분기 처리가 품질 면에서 최적입니다."

### C-1. 대형 금융 STT 기능 요건 상세 논의

| 항목 | 내용 |
|------|------|
| **출처** | KB FCC (금융그룹 7개 계열사 통합) |
| **데이터 소스** | 패턴DB + 도메인 지식 (전사 회의록 없음) |

**Q1. 실시간 스트리밍 STT가 기술적으로 가능한가요? 파셜(partial) 결과를 상담사가 바로 볼 수 있어야 합니다.**

| 항목 | 정답 |
|------|------|
| 기본 답변 | 가능하다. EC STT 엔진은 실시간 스트리밍 모드를 지원하며, 파셜(Partial) 결과를 상담사 화면에 즉시 표시할 수 있다. |
| RT 게이트웨이 | 실시간 파셜 기능은 **RT 게이트웨이**를 통해 구현되며, 음성기술팀의 스트리밍 모델이 선행 개발되어야 한다 (패턴DB #28: "RT 게이트웨이에서 파셜 기능 개발하려면 음성기술팀 스트리밍 모델이 먼저 돼야 하는 거 아닌가요?") |
| 아키텍처 | RX/TX 양방향 스트리밍으로 상담사/고객 음성 동시 처리. 100채널 기준 서버 2대 구성 (NH투자증권 사이징 참고: "100채널인데 RSTS 합쳐서 200채널 때문에 장비는 2대") |
| 실운영 실적 | 유플러스 자체 고객센터(4,000~5,000석)에서 실시간 STT를 운영 중. 어드바이저 화면에서 고객/상담사 발화가 실시간 텍스트로 노출되는 구조 (한화 회의록에서 시연 언급) |

**Q2. STT 결과를 DB에 저장할 수 있어야 합니다. 저장 기간과 자동 삭제 정책은?**

| 항목 | 정답 |
|------|------|
| DB 저장 | STT 결과는 API를 통해 DB에 적재 가능. 콜키 기반으로 조회 가능 (패턴DB 요건정의 #14, 기능/성능 #17) |
| 암호화 | 금융권 필수 요건으로 DB 저장 시 AES-256 암호화 적용 가능 (패턴DB 보안/규제 #10: "암호화 저장은 기본인데, 어떤 암호화 방식을 쓰나요?") |
| 보관기간/자동삭제 | 계열사별로 보관 기간 차등 설정 가능 (예: 5년, 7년). 금감원 규제에 따른 보관 의무 기간 준수 |
| 개인정보 처리 | NER(개체명 인식) 적용하여 계좌번호, 이름 등 개인정보 자동 태깅 및 마스킹 가능 (한화 회의록: "계좌 정보 이름 같은 경우에는 NER을 적용해서 태깅이 되게 되어 있습니다") |

**Q3. STT 학습 후 배포까지 자동화가 가능한가요?**

| 항목 | 정답 |
|------|------|
| 학습 도구 | 웹 기반 관리 도구에서 사전 관리 및 학습 수행 가능. 금융 용어/상품명은 파일 업로드 후 버튼 클릭으로 적용 (한화 회의록: "파일을 업로드하고 버튼을 클릭해서 적용까지 하는 기능은 웹 형태로 제공") |
| 자동화 수준 | LLM 기반 품질 개선 자동화 툴(AI-in-the-loop) 적용. 오분류 데이터 자동 샘플링 → LLM 정답 판정 → 벡터 DB 임베딩까지 자동화. **기존 8명 투입 → 현재 0명(자동화 완료), 5,760분 → 40분으로 단축** (한화 회의록) |
| 인식률 측정 | 학습 후 파일 단위 인식률 측정 기능 웹에서 제공. 정답지와 비교하여 즉시 확인 가능 |
| 배포 방식 | 일 단위 배치로 KMS/사전 업데이트 자동 반영. 고객사와 협의하여 학습 주기 결정 |

**Q4. 계열사마다 다른 SSO를 전부 연동 가능한가요?**

| 항목 | 정답 |
|------|------|
| 가능 여부 | 기술적으로 가능. 다만 **계열사 수만큼 개별 작업 필요** (패턴DB #30: "SSO가 계열사마다 다르다고 하는데, 그러면 계열사 수만큼 작업해야 하는 건가요?") |
| 공수 산정 | 계열사별 SSO 프로토콜(SAML, OAuth, OIDC 등)이 다를 수 있으므로, 사전 조사 → 개별 연동 → 테스트 순으로 진행. 계열사당 약 1~2주 소요 |
| 주의점 | 7개 계열사 병렬 진행하더라도 최소 6~8주 소요. '한 번에 된다'고 절대 말하지 않는다 |

**Q5. KB CICD 파이프라인 연동 규격은?**

| 항목 | 정답 |
|------|------|
| 연동 방식 | KB 내부 CICD 규격에 맞춘 연동 필요 (패턴DB #29: "KB CICD 파이프라인에 연동하려면 어떤 규격을 맞춰야 하나요?") |
| 아키텍처 적합성 | 쿠버네티스/MSA 기반 아키텍처로 CICD 파이프라인 연동에 유리 (한화 회의록: "쿠버네티스 기반으로 솔루션이 구축되어 있고 MSA 아키텍처 기반") |
| 사전 협의 | KB 보안팀과 이미지 스캐닝 정책, 배포 승인 프로세스, 형상관리 규격 등을 사전 확인 필요 |

**함정 대응 가이드**

- **RT 게이트웨이 파셜 기능 일정 리스크**: "파셜 기능은 음성기술팀의 스트리밍 모델 개발이 선행되어야 하며, 현재 개발 진행 중입니다. 전체 일정에 이 선행 작업 기간(약 2~3개월)을 반드시 포함해야 합니다. 대안으로 배치 모드 우선 적용 → 스트리밍 순차 전환이 가능합니다."
- **계열사별 SSO 공수 과소평가**: "7개 계열사 각각 SSO 프로토콜이 다를 수 있어 개별 연동이 필요합니다. 병렬 진행하더라도 최소 6~8주를 잡아야 하며, 사전에 각 계열사 인증 담당자와 킥오프가 필요합니다."

### C-2. 클라우드 콜봇 연동 아키텍처 설계

| 항목 | 내용 |
|------|------|
| **출처** | NH투자증권 |
| **데이터 소스** | NH투자증권 회의록 직접 인용 (2026.03.27) |

**Q1. 중계서버가 꼭 필요한가요? 시스코가 폐쇄적이라서 그런 건가요?**

| 항목 | 정답 |
|------|------|
| 결론 | **필요하다.** 시스코 PBX가 폐쇄적이어서 직접 연동이 어렵기 때문에 **VBG(Voice Bot Gateway)**를 통해 연동해야 한다 |
| VBG 스펙 | 원래 RT 게이트웨이로 계획했으나, 콜봇 연동을 위해 보이스봇 게이트웨이(VBG)로 변경. 사이징 8코어 → **16코어로 상향** (회의록: "보이스봇 게이트웨이로 들어가야 된다고 얘기가 나와서 바꾸긴 했어. 사이징도 키웠어. 8코어에서 16코어로") |
| 라이센스 | VBG 라이센스비(석당 라이센스)가 별도 필요. 초기 견적에서 누락되었던 항목. **채널당 약 55~60만원** (회의록: "55만 원 하려고 그랬는데 60만 원 하시더라") |
| 아키텍처 | 시스코 CTI → VB 게이트웨이 → STT/TTS 엔진 → LLM → TTS → 고객 응답 |

**Q2. 개인정보 필터링은 어디서 하나요? 콜봇이 직접 하나요?**

| 항목 | 정답 |
|------|------|
| 필터링 위치 | **어드바이저 솔루션 내 STT 후처리 단계**에서 수행. 콜봇이 직접 하는 것이 아닌 3단계 구조: STT 인식 → NER 마스킹 → DB 저장 시 암호화 |

**Q3. FAQ와 RAG 중 어떤 방식으로 처리할지 어떻게 결정하나요?**

**Q1. 기존 STT(마하겐/엔솔루) 교체 시 연동은 어떻게 되나요?**

| 항목 | 정답 |
|------|------|
| 교체 가능성 | 기존 레거시 인프라 변경 없이 STT 도입 가능. 게이트웨이만 추가하면 됨 (한화 회의록: "기존의 레거시 인프라 환경에 대한 변경 없이 STT를 도입할 수 있다") |

**Q2. RAG는 MongoDB vs 엘라스틱서치 중 어떤 걸 쓰나요?**

| 항목 | 정답 |
|------|------|
| 현재 사용 | **AWS MongoDB Atlas** 기반 RAG 운영 중 (한화 회의록: "RAG 같은 경우에는 AWS 몽고DB 아틀라스 기능을 쓰고 있고") |

| 온프레미스 대안 | 금융권 보안 정책상 외부 클라우드 DB 사용이 어려우면, 온프레미스 Elasticsearch + 벡터 플러그인 또는 Milvus/Weaviate 등 오픈소스 벡터 DB 대안 가능 |

**Q4. LLM 모델 교체 시 재검증에 얼마나 걸리나요?**

**Q5. 생성형 AI 부분만 빼고 STT/TTS/TA만 구축할 수 있나요?**

### D-1. STT/TTS 기술 견적 협의

**Q1. 왜 이렇게 비싸냐?**

| 항목 | 정답 |
|------|------|
| 견적 성격 | 원래 **리스트 프라이스(표준 금액)**로 넣은 가견적. 실제 입찰 시 할인 적용 필수 (회의록: "진짜 그냥 가 견적이라고 해서 리스트 프라이스로 그냥 표준 금액으로 넣었었어요. 제안하실 때 판단하셔서 할인하시라고") |
| TTS 라이센스 적정가 | **사이트 라이센스**: 2,000~2,500만원, **채널 라이센스**: 채널당 40~50만원 (회의록: "사이트 라이센스 한 2천만 원에 채널 라이센스 채널당 한 50만 원 40만 원 이 정도") |
| 할인 후 금액 | 사이트 라이센스 1,500만원 + 채널 라이센스 = 약 **7,500만원 수준** (75% 할인 적용) (회의록: "2500 그러면 7500 이것도 너무 비싼가요?... 아니요. 일단 네 아름답네요") |
| 대응법 | "현재 보신 금액은 표준 리스트 가격입니다. 실제 제안 시에는 채널 규모와 계약 조건에 따라 상당한 할인이 적용됩니다. 채널당 40~50만원 수준으로 조정 가능합니다." |

**Q2. TTS 하는데 사람이 5명이나 들어간다고?**

| 항목 | 정답 |
|------|------|
| 원래 구성 | 라피치 표준: 중급 2명 + 초급 3명 = 5명. 보수적 산정 (회의록: "이것도 라피치한테 그냥 일반적인 거 주세요 그러면 이렇게 줄 거예요. 보수적으로 잡는 거죠. 연동이 뭐가 있을지 모르고 커스터마이징이 뭐가 있을지 모르니까") |
| 조정 방안 | **중급 1명 + 초급 3명 = 4명(4 M/M)으로 축소 가능** (회의록: "일단 중급으로 4멤버스로 잡아도 되지 않아요") |
| PL 비용 처리 | PL 단가를 별도 항목으로 넣지 않고, 프로젝트 관리비에 흡수 (회의록: "PL 비용이라고 따로 받지는 않고 작업의 범위에 대해서 책정을 하고 거기에다가 사람들을 투입") |
| 견적서 구조 | 작업 범위(과업 난이도)별 초급/중급으로 분리. PL은 인력 투입 프로필에만 명기 |

**Q3. GPU 없이 TTS가 가능한가요?**

| 항목 | 정답 |
|------|------|
| 결론 | **가능.** EC TTS는 뉴럴 TTS이지만 GPU 대신 CPU(고코어)로 운영 가능 |
| CPU 사양 | **최소 24코어** 필요 (원래 32코어 → 채널 축소로 24코어로 조정). 메모리 16GB (회의록: "24코어는 있어야 될 것 같고요") |
| GPU vs CPU 비교 | GPU: 레이턴시 감소, 음성 품질 약간 우위. CPU: **8kHz/16kHz 콜센터 환경에서 체감 차이 미미** (회의록: "8k 16k 정도면 제 귀로는" 차이 못 느낌). "업계에서 워낙 TTS를 가성비를 중요하게 생각" |
| 전략적 포인트 | **GPU 미사용이 오히려 장점으로 어필 가능** (회의록: "TTS GPU 사용해요 물어봤을 때 안 사용하는 게 더 좋으니까") |
| 클라우드 유의 | 퍼블릭 클라우드(AWS) 배포 시 TTS 권장 인스턴스가 GPU 인스턴스일 수 있음. GPU 불요 옵션으로 별도 사이징 요청 필요 |

**Q4. 패키지 형태로 묶어서 제공할 수 있나요?**

| 항목 | 정답 |
|------|------|
| 현재 구조 | STT와 TTS는 **별도 엔진**으로 개발. 완전 통합 패키지는 어려움 (회의록: "이거를 묶기는 어렵지 않을까요? 애초에 다른 엔진으로 구성된 요소고") |
| 패키지화 방향 | **세트 구성표** 제작 가능 (회의록: "세트를 하나 만들었으면 좋겠어요. 콜봇용, 대화록용, TTS만 붙이는 용... 세트를 만들어 놓으면 착착착 갖다 넣으면 되니까") |
| 세트 유형 | (1) 콜봇 세트: STT + TTS + VBG + 콜봇 엔진 (2) 대화록 세트: STT + TA + 어드바이저 (3) TTS 단독 세트: TTS + IVR 연동 |

**Q5. 외부 LLM 연동 vs 리얼타임 API, 어떤 게 나은가요?**

| 항목 | 정답 |
|------|------|
| 두 가지 옵션 | (1) 외부 LLM(GPT/Gemini) 연동: **AWS 기반**, 표준 구성, RAG 구현 용이 (2) 리얼타임 API(Speech-to-Speech): **Azure 기반**, 초저지연, 아직 실험적 |
| 회의록 근거 | "하나는 외부 GPT든 제미나이든 연동하는 버전... 두 번째는 리얼타임 API로 만든 거" |
| 주의사항 | 리얼타임 API에 STT가 왜 들어가는지 내부적으로도 혼선 있음 (회의록 21:43~22:33). 두 버전을 동시에 준비할 수 없어 순차 진행 |
| 대응법 | "두 가지 아키텍처를 모두 견적으로 제출합니다. 안정성 중시면 외부 LLM 연동 버전, 초저지연이 최우선이면 리얼타임 API 버전을 권장합니다." |

**함정 대응 가이드**

- **경쟁사 가격 (리드스피커 TTS 1,700만원)**: 고객이 이미 가격을 알고 있음 (회의록: "리드 스피커나 이런 데 뉴럴 TTS 완전 싸죠 1700만원이에요"). 대응: "리드스피커는 사이트 라이센스 없이 채널당 55만원 x 30채널 = 1,650만원 구조입니다(에스원 사례). 저희는 채널당 40~50만원으로 리드스피커보다 경쟁력이 있으며, 자체 학습/유지보수가 가능한 점이 핵심 차별점입니다."
- **견적 항목 누락**: VBG 라이센스(채널당 55~60만원), STT/TTS 관리 포탈, 개발/운영 서버 분리, PL 비용 처리 방식 등을 빠짐없이 확인. "가견적이지만, 모든 구성 요소를 포함한 완전한 견적을 제출하는 것이 전문성을 보여주는 핵심입니다."

### D-2. 멀티벤더 솔루션 비교 협의

| 항목 | 내용 |

| **데이터 소스** | 패턴DB + 도메인 지식 (전사 회의록 없음) |

**Q1. 시카스(SICAS)는 검토한 적 있나요?**

| 항목 | 정답 |
|------|------|
| 검토 이력 | Bright Pattern 등 대안 벤더 검토 이력 존재 (패턴DB 도입 #18: "Bright Pattern은 검토한 적 있나요?") |
| 시카스 한계 | 국내 CTI 시장 점유율은 있으나, KB FCC 규모(7개 계열사 통합, 대규모 동시 채널)에 맞는 엔터프라이즈 레퍼런스 제한적 |
| 배경 이슈 | "제네시스 EOS가 언제예요? 저희 계약은 언제까지 가능한 거죠?" (패턴DB 도입 #17) → 제네시스 EOS 대응이 CTI 교체의 핵심 동인 |
| 대응법 | "시카스도 검토 대상에 포함시킬 수 있습니다. 다만 7개 계열사 통합 운영 규모의 레퍼런스와 기술지원 체계를 확인 후 비교표에 추가하겠습니다." 절대 '검토할 가치가 없다'고 말하지 않는다 |

**Q2. CTI가 바뀌면 통계, 소프트폰, PDS 전부 바뀌어야 하는 건가요?**

| 항목 | 정답 |
|------|------|
| 결론 | **맞다. CTI 교체 시 연쇄 변경이 발생한다** (패턴DB #25: "CTI가 바뀌면 통계도 바뀌어야 하고, 소프트폰도 바뀌어야 하고, PDS도 다 바뀌는 거 아닌가요?") |
| 영향 범위 | CTI 교체 → (1) 통계 시스템 재구축, (2) 소프트폰 클라이언트 교체, (3) PDS 교체, (4) 녹취 연동 변경, (5) IVR 연동 변경, (6) CRM/상담 AP 인터페이스 수정 |
| 비용 영향 | CTI 교체 비용 자체보다 연쇄 변경에 따른 **총 전환 비용(TCO)**이 더 크다 |
| 대응법 | "CTI 교체는 단독 항목이 아니라 생태계 전체의 전환입니다. 각 항목별 전환 비용과 일정을 포함한 TCO 분석을 제공하겠습니다." |

**Q3. PDS 단가가 3.5만원에서 5.8만원으로 올라간다고요? (65% 인상)**

| 항목 | 정답 |
|------|------|
| 단가 변동 | 테라/라피치 PDS → 샌드박스 PDS 교체 시 석당 3.5만원 → 5.8만원, **약 65% 인상** (패턴DB 비용 #9) |
| 인상 정당화 | (1) 신규 PDS는 AI 기반 예측 다이얼링 기능 포함 (2) 콜 블렌딩(인/아웃바운드 통합) 지원 (3) 클라우드 네이티브 아키텍처로 확장성 우위 (4) 기존 PDS는 EOS 임박 |
| TCO 비교 | "단가만 비교하면 65% 인상이지만, AI 예측 다이얼링으로 연결률이 개선되어 실질 필요 채널 수가 줄어듭니다. **캠페인당 총 비용**으로 비교하면 오히려 비용 효율적입니다." |

**Q4. 7개 솔루션 비교표를 만들 수 있나요?**

| 항목 | 정답 |
|------|------|
| 비교 대상 | CTI, STT, TTS, TA, 콜봇, PDS, 어드바이저(Agent Works/FC 2.0) 등 7개 영역 (패턴DB #26: "에이전트 웍스가 뭔가요? FC 2.0이라고 하던데 기존 TA랑 어떻게 달라요?") |
| 비교 기준 | 기능, 가격, 레퍼런스, 확장성, 유지보수, 벤더 안정성, EOS 시기 |
| 구조 제안 | **1페이지 요약(경영진용) + 상세 별첨** 구조. 핵심 5개 + 참고 2개로 구분하여 의사결정에 집중 |
| 시간 관리 | "비교표 초안 3일 내 제공, 경영진 피드백 후 5일 내 최종본 제출" |

**Q5. TA 비용이 왜 이렇게 많이 나오나요?**

| 항목 | 정답 |
|------|------|
| 질문 배경 | TA 사용 상담사가 적은데 비용이 과다 (패턴DB 비용 #10: "TA 쓰는 상담사가 얼마 없는데 왜 TA 비용이 이렇게 많이 나와요?") |
| 비용 구조 | TA 비용 = 엔진 라이센스(서버 기반) + STT 전사 비용 + 분석 규칙/모델 개발비. **사용 상담사 수가 아닌 분석 대상 통화 건수에 비례** |
| 절감 방안 | (1) STT 인프라 공유로 이중 투자 방지 (2) 전수 분석이 아닌 샘플링으로 시작 (3) 기본 분석 규칙은 패키지 제공, 커스텀만 추가 개발 |
| 대응법 | "TA 비용이 높게 보이는 이유는 전수 분석 기준으로 산정되어 있기 때문입니다. 샘플링 분석으로 시작하면 초기 비용 50% 이상 절감 가능합니다." |

**함정 대응 가이드**

- **7개 벤더 비교표 시간 압박**: "핵심 5개 벤더를 우선 비교하고 나머지 2개는 참고로 별첨합니다. 3일 내 초안 제출 가능합니다."
- **PDS 교체 단가 65% 인상 정당화**: "단가만 비교하지 마시고, AI 예측 다이얼링으로 인한 연결률 개선 효과를 포함한 TCO 비교를 함께 보셔야 합니다. 3년 TCO로 비교하면 기존 대비 약 20% 증가 수준이며, 기능 향상분을 감안하면 합리적인 투자입니다." 반드시 숫자로 시뮬레이션 제공.

### D-3. 구독형 vs 구축형 비용 비교 협의

| 항목 | 내용 |
|------|------|
| **출처** | 하나투어/소노 시니어 (여행/레저 업종) |
| **데이터 소스** | 패턴DB + 도메인 지식 (전사 회의록 없음) |

**Q1. 풀 구축 예산 10억 중반대가 가능한가요?**

| 항목 | 정답 |
|------|------|
| 가능 여부 | 풀 AICC 구축(STT + TTS + 어드바이저 + 콜봇 + TA + 오토QA)은 **10억 중반대(14~16억)로 가능하지만 타이트** |
| 비용 구성 | STT 엔진/서버(3~4억) + TTS(2~3억) + 어드바이저/KMS(2~3억) + 콜봇(3~4억) + TA/QA(1~2억) + 인프라(1~2억) |
| 절감 방안 | (1) GPU 미사용 TTS로 인프라 절감 (2) 생성형 AI는 SLM(온프레미스)으로 외부 API 비용 제거 (3) 오토QA는 2단계 분리 |
| 권장 | "10억 중반이면 1/2단계 분리 권장. 1단계(7~8억): STT + 어드바이저 + TA, 2단계(5~6억): 콜봇 + 오토QA. 각 단계별 ROI를 확인하며 진행하면 리스크가 줄어듭니다." |
| 패턴DB 참고 | "구축형으로 한다고 하면 수억 정도 든다고 생각하면 되나요?" (패턴DB 비용 #17) |

**Q2. 벤더 종속성(Lock-in) 리스크는 어떻게 대응하나요?**

| 항목 | 정답 |
|------|------|
| 락인 방지 설계 | (1) **모듈형 아키텍처**: 각 엔진(STT, TTS, LLM, RAG)을 독립 모듈로 구성, 개별 교체 가능. "넣고 빼고 가능한 형태" (한화 회의록) (2) **표준 API 인터페이스**: REST API/gRPC 기반 (3) **데이터 소유권 보장**: 학습 데이터, KMS 콘텐츠, 분석 결과 모두 고객 소유 |
| LLM 멀티모델 | GPT + Gemini 하이브리드 운영 검증 완료. 온프레미스 SLM(Qwen 32B, GPT OSS 120B)으로 외부 의존도 제거도 가능 (한화 회의록) |
| 계약 조건 | 소스코드 에스크로, 유지보수 SLA, 데이터 이관 지원 등 계약서에 명시 권장 |
| 대응법 | "벤더 종속 우려는 저희도 중요하게 생각합니다. 모듈형 아키텍처 + 표준 API + 데이터 소유권 보장 3가지로 이를 해소합니다. 계약서에도 명시합니다." |

**Q3. 시즌마다 내용이 바뀌는데 KMS 업데이트는 어떻게 하나요?**

| 항목 | 정답 |
|------|------|
| 업데이트 방식 | (1) 웹 UI에서 현업 담당자가 직접 업로드 → 자동 청킹/임베딩/반영 (2) 일 단위 배치 자동 동기화 (한화 회의록) (3) 긴급 시 실시간 반영 모드(수동 트리거) |
| 시즌 전환 | 시즌 전환 2주 전: 신규 상품 정보 사전 등록 → 시작일: 배치 자동 전환 → 이전 시즌 콘텐츠 비활성화(아카이브) |
| 신규 시나리오 | "정말 새로운 업무 서비스가 들어온다면 조금 어려운 거죠. 1년에 몇 개 이상은 무상으로 계약을 아예 해놓고 진행" (한화 회의록). IT 담당자와 현업이 짝을 이뤄 시나리오 관리 |
| 대응법 | "시즌별 KMS 업데이트는 현업이 웹 포탈에서 직접 수행 가능합니다. 다만 시즌 전환 시 누락 방지를 위한 운영 SOP가 필요합니다. 저희가 시즌별 콘텐츠 체크리스트와 업데이트 프로세스를 함께 설계해 드리겠습니다." |

**Q4. 온프레미스 필수인데 SaaS를 제안하는 이유는?**

| 항목 | 정답 |
|------|------|
| 하이브리드 제안 | 온프레미스 필수 요건을 존중하되, 비용 효율성을 위한 하이브리드 구성 제안 |
| 온프레미스 대상 | STT, TTS, TA (음성 데이터 처리 = 개인정보) → 반드시 온프레미스 |
| SaaS 가능 대상 | LLM API 호출, KMS 관리 포털, 대시보드/리포팅 → SaaS로 비용 절감 |
| 비용 비교 (3년) | 구축형: 초기 15억 + 유지보수 연 2.5억 x 3년 = **22.5억**. 구독형: 월 2,000만원 x 36개월 = **7.2억**(단, 채널 확장 시 증가). **손익분기점 약 4~5년** |
| 여행/레저 특수성 | 시즌별 트래픽 변동이 큰 업종은 구독형(사용량 기반)이 비용 효율적. 성수기만 채널 확장 가능. "클라우드 테스트해보고 반응이 괜찮으면 구축형으로 넘어가야 될 것 같은데, 결이 엄청 다르지는 않죠?" (패턴DB 도입 #23) |

**함정 대응 가이드**

- **벤더 락인 우려**: 절대 회피하지 않는다. "맞습니다. 벤더 종속은 실제 리스크입니다. 그래서 저희는 모듈형 아키텍처 + 표준 API + 데이터 소유권 보장 3가지로 이를 해소합니다. 계약서에도 이 조항을 명시합니다."
- **시즌 KMS 관리 난이도 과소평가**: "기술적으로는 단순하지만, 운영 프로세스가 없으면 시즌 전환 시 오래된 정보가 고객에게 안내되는 사고가 발생합니다. 반드시 시즌별 KMS 업데이트 SOP를 함께 수립해야 합니다. 한화손보에서도 '1년에 몇 개 이상은 무상으로 계약'하는 방식을 채택하고 있습니다."

#### Q1. 구독형이 월 40만원이면 되는 건가요?

| 항목 | 정답 (패턴DB + 범용) |
|------|----------------------|
| 핵심 포인트 | 구독형 월 사용료 40만원은 기본 요금 기준이며, 이것은 **채널 수/동시 통화 수에 따라 달라짐**. 아웃바운드 콜봇의 경우 PDS(Predictive Dialing System) 연동이 필요하며, PDS 장비/소프트웨어 비용은 별도 (패턴DB 비용/라이센스 #15: "구독형이라는 게 월 사용료로 보면 되겠네요") |
| 추가 비용 요소 | 별도 클라우드 구축 비용 발생 가능 (패턴DB #16: "별도 클라우드를 구축해야 된다고 또 구축 비용이 또 드는 거겠네요"), STT/TTS 라이센스, 시나리오 커스터마이징, 기관계 연동 개발비 별도 |

#### Q2. 파일럿이 2~3개월씩 걸리나요?

| 항목 | 정답 (패턴DB + 범용) |
|------|----------------------|
| 핵심 포인트 | 파일럿(PoC) 기간은 업무 범위와 시나리오 복잡도에 따라 상이. 단순 시나리오(고지성 아웃바운드)는 2개월 이내 가능하나, 기관계 연동 + 복합 시나리오 포함 시 2~3개월 필요 (패턴DB 도입/구축 #22) |
| 기간 산정 근거 | 한화손보 회의록 기준: 음성봇 자체 구축은 "기본적으로 6개월"이며, STT/TTS만 적용해도 "4개월은 봐야" 함. PoC는 전체 구축의 축소판이므로 최소 2개월은 현실적 |
| 단계별 일정 | (1) 요건 분석/시나리오 설계: 2~3주 (2) 시나리오 개발/STT-TTS 연동: 3~4주 (3) 테스트/튜닝: 2~3주 (4) 파일럿 운영/결과 분석: 2~3주 |

#### Q3. 콜 블렌딩이 되나요?

| 항목 | 정답 (패턴DB + 범용) |
|------|----------------------|
| 핵심 포인트 | 콜 블렌딩(상담사가 콜봇 통화 중간에 개입)은 기술적으로 가능. 패턴DB 기능 #21: "상담사가 중간에 끼어들 수도 있는 건가요?" — 콜봇이 처리하다가 복잡한 케이스 발생 시 상담사로 자동 전환하는 구조 |

#### Q4. 파일럿 후 구축형 전환이 쉬운가요?

| 항목 | 정답 (패턴DB + 범용) |
|------|----------------------|
| 핵심 포인트 | 클라우드 파일럿과 구축형(온프레미스)은 **결이 다름을 솔직히 설명**해야 함 (패턴DB 도입/구축 #23: "클라우드 테스트해보고 반응이 괜찮으면 구축형으로 넘어가야 될 것 같은데, 결이 엄청 다르지는 않죠?") |
| 전환 시 차이점 | (1) 인프라: 클라우드 API 호출 → 온프레미스 서버 구축(GPU/CPU 장비 발주 1~2개월) (2) 보안: 금융권 특히 온프레미스 필수(보이스 데이터=개인정보) (3) 비용: 구축형은 수억 규모(패턴DB #17), 장비비+라이센스+구축비 (4) 시나리오/비즈니스 로직은 재활용 가능하나, 연동 인터페이스는 재구축 필요 |
| 전환 전략 | 파일럿에서 검증한 시나리오/학습 데이터/튜닝 결과는 구축형에도 이관 가능. 단, 아키텍처 차이를 과소평가하면 안 됨 |

#### Q5. 경영진이 1~2주 내를 원합니다.

| 항목 | 정답 (패턴DB + 범용) |
|------|----------------------|
| 핵심 포인트 | 1~2주는 **기술적으로 불가능**한 일정. 솔직하게 설명 필요 (패턴DB 도입/구축 #25: "경영진 미팅에서 빨리 해라. 빨리의 관점은 1~2주 내입니다.") |
| 현실적 대안 | (1) 1~2주 내 가능한 것: 구독형 데모 환경 셋업 + 기본 시나리오 시연 → "2주 내 경영진 데모"는 가능 (2) 실제 파일럿: 최소 2개월 (3) 데모 vs PoC vs 본구축의 차이를 경영진용 1장 요약으로 정리하여 제공 제안 |
| 대응 화법 | "경영진께 2주 내에 **동작하는 데모**를 보여드리는 것은 가능합니다. 다만 실제 CJ 업무 데이터 연동 파일럿은 요건 분석부터 시작해야 하므로 최소 2개월이 필요합니다. 데모에서 방향성 확인 → 파일럿 착수 결재, 이 2단계 접근을 제안드립니다." |

**핵심 함정 대응**:
- **비현실적 시간 압박**: "1~2주"라는 요구에 "해보겠습니다"라고 하면 신뢰 상실. 대신 1~2주 내 할 수 있는 것(데모)과 할 수 없는 것(파일럿)을 명확히 구분하여 제시
- **구독→구축 아키텍처 차이 과소평가**: "똑같습니다"라고 하면 나중에 구축형 전환 시 추가 비용/일정 발생으로 불신. 차이를 미리 투명하게 설명하되, 시나리오/데이터 자산은 재활용 가능하다는 점 강조

### E-2. 보이스봇 요건정의 / RFP 지원 (에스원)

#### Q1. 장비 점검 안내 아웃바운드를 보이스봇으로 돌릴 수 있나요?

| 미인증 고객 처리 | 전산에 등록되지 않은 고객은 상담사로 자동 전환 (회의록: "신분 확인이 안 되는 경우... 그냥 상담사로 넘겨버릴 겁니다") |

#### Q2. 기관계 연동이 가능한가요?

| 항목 | 정답 (회의록 기반) |
|------|-------------------|
| 핵심 포인트 | 필요하며, **콜 상담 시스템**(현재 ERP 재개발 중)과 연동이 핵심. 보이스봇이 수집한 정보를 콜 상담 시스템에 저장하고, 해당 계약번호에 매핑된 출동 차량으로 자동 전송하는 구조 |
| 연동 대상 시스템 | (1) 콜 상담 시스템 — 고객 정보 팝업(전화번호 매핑), 접수 내용 저장 (2) 출동 시스템(VO 시스템) — 요약된 접수 내용 전송, 출동 차량 배치 |

| 관제 시스템 연동 | 불필요 — "관제 시스템하고는 크게는 없을 것 같다" (회의록). 콜 상담 시스템 내에서 VO 시스템으로 전달하는 기존 프로세스 활용 |
| ERP 재개발 유의사항 | 콜 상담 시스템이 ERP 재개발 범위에 포함될 수 있으므로, 연동 인터페이스 사전 협의 필요 (회의록: "ERP 개편하는 거예요. 영역 밖이라서") |

#### Q3. 전화 미응답 시 재발신 로직이 있나요?

| 항목 | 정답 (범용 + 회의록 참고) |

| 핵심 포인트 | 아웃바운드 보이스봇의 재발신 로직은 PDS(또는 ACS) 연동으로 구현. 에스원의 경우 현재 PDS 미사용(상담사 직접 발신), 보이스봇 도입 시 PDS 도입 또는 ACS 기반 자동 발신 구조 필요 |
| 재발신 설계 요소 | (1) 미응답 시 재시도 횟수(보통 3~5회) (2) 재시도 간격(30분~1시간) (3) 시간대 제한(오전 9시~오후 6시) (4) 부재중 후 SMS 발송 옵션 |

#### Q4. 텍스트 요약도 같이 가능한가요?

| 항목 | 정답 (회의록 기반) |

#### Q5. 시니어 고객 비표준 발화에도 STT가 잘 되나요?

| 항목 | 정답 (회의록 + 범용) |
|------|---------------------|
| 핵심 포인트 | 시니어 고객의 비표준 발화는 STT의 알려진 약점. 에스원 회의록에서 이미 이 문제를 인지하고 **신분 확인 단계를 "예/아니오"로 단순화**하는 설계를 채택 |
| 대응 전략 (회의록) | "번호 같은 거는 말하는 게 STT가 좀 잘 안 맞을 수도 있으니까 예를 들어 예 아니오 정도로 누구 어떤 고객이 맞으십니까 하는 거 예 아니오 정도로만 확인" |
| STT 성능 기준 | 금융 특화 학습 후 93~96% 인식률 달성 가능 (한화손보 PoC 기준: KB손보 95%, 상담원 98%, 고객 92%). 시니어 고객은 고객 측 인식률(92%)보다 낮을 수 있으므로, 시나리오를 선택형/확인형으로 설계하여 보완 |
| RFP 반영 사항 | STT 인식률 요건은 "일반 고객 기준"과 "시니어 고객 기준"을 분리하여 정의할 것을 권장. 시니어 특화 사전(방언, 줄임말) 학습 추가 |

**핵심 함정 대응**:
- **고객 업무 프로세스 이해 부족**: 에스원의 기기점검 VOC 흐름을 정확히 이해해야 함 — (1) 1번 메뉴 전진배치 (2) 신분확인은 예/아니오 (3) 장비 종류는 카메라/카드리더기/지문인식기/소프트웨어(근퇴 프로그램) (4) 출동 차량은 장비 종류 무관 동일 (5) 접수경로 코드 신규 생성 필요
- **RFP 스펙 과다/과소**: RFP에 포함해야 할 항목 — AI 엔진 스펙, STT/TTS 상세(텍스트 서머리 포함), 하드웨어 스펙, 콜 상담 시스템 연동 인터페이스. 에스원이 직접 쓰되, 솔루션사가 세부 스펙(AI 엔진, STT/TTS, 하드웨어)을 채워서 제공해야 함 (회의록: "좀 잘 써서 주시면은 제가 좀 최대한 반영을 해서 RFP를 잘 다시 제작해서")
- **Phase 2 확장 인지**: 1차는 기기점검이지만, 2차에 365코너 ATM 장애/일정 연장/장비 해지 등 추가 예정 — 확장성 있는 아키텍처 제안 필요
- **콜 저감 목표**: 보수적 40% (3년), 공격적 60% 가능 (회의록: "40% 넘길 수 있을 것 같아요... 공격적 60%까지"). 월간 볼륨: 해지 코드 최대 10,000건+, 기타 약 4,000건

#### Q4. CER 측정 기준을 통일해야 해요.

|------|--------------------------------|

| 합의 필요 항목 | (1) 측정 지표: CER or WER (한국어 특성상 CER 권장) (2) 정답지 생성 방법: 자동 vs 수동 전사 (패턴DB 기능/성능 #18: "정답지 생성 기능이 있어야 하고, 거기서 인식률 계산도 되어야") (3) 샘플 선정 기준: 랜덤 vs 난이도별 층화 (4) 목표치: 일반 통화 93% 이상, 금융 용어 포함 90% 이상 등 (5) 제외 조건: 잡음, 3초 미만 발화 등 |
| 리스크 | CER 기준 미통일 시 벤더별 측정 방식이 달라져 PoC 불합격 사태 발생 가능. **착수 전 측정 프로토콜 문서화 필수** |

- **CER 기준 미통일**: PoC 착수 전 측정 프로토콜 합의 문서를 만들어야 함. 벤더마다 유리한 방식으로 측정하면 비교 불가능해짐
- **GPU 조달 리스크**: "H200 없으면 못합니다"가 아니라, 대안 GPU + 아키텍처 최적화로 해결 가능하다는 것을 보여줘야 기술 역량 신뢰 확보

- **임원 시연 리스크**: 라이브 시연 실패 시 대비책 필수. 사전 녹화 데모 + 실시간 시연 병행, 최악의 경우 대응 멘트 준비 ("이 케이스는 잡음 환경에서의 인식이며, 정상 환경에서는 95% 달성" 등)

#### Q4. CER 현재 수치와 목표치는?

#### Q5. 개선 일정은 어떻게 되나요?

| 항목 | 정답 (대신증권 회의록 + 범용) |
|------|------------------------------|
| 핵심 포인트 | **낙관적 일정은 재실망 리스크**. 보수적으로 잡되, 마일스톤별 점검 포인트를 제시 |

**핵심 함정 대응**:
- **주관적 피드백에 기술적 원인 분석**: "자연스럽지 못했다"는 피드백을 그대로 두지 말고, 샘플 콜 확보 → 건별 원인 분석(인식률/TTS/레이턴시/대화흐름) → 정량화된 개선 계획으로 전환
- **낙관적 개선 일정 시 재실망**: "2주면 됩니다"라고 했다가 안 되면 신뢰 완전 상실. 보수적 일정(8주) + 2주 단위 마일스톤 + 중간 점검으로 투명하게 관리. "최선의 경우 4주, 현실적으로 6~8주"와 같이 범위로 제시

#### Q2. 다양한 샘플링 주파수(8kHz, 16kHz, 48kHz) 지원 실제 가능한가요?

| 항목 | 정답 (참고) |
|------|-----------|
| **핵심 포인트** | STT 엔진은 입력 음성의 샘플링 주파수에 따라 성능이 달라짐. **모델이 학습된 주파수와 입력 주파수가 일치해야 최적 성능** |
| **실제 미팅 근거** | 패턴DB 쿠팡: *"배치 STT 처리에서 다양한 샘플링 주파수 지원 관련해서 매뉴얼에는 이런 내용은 없긴 하거든요."* |
| **주파수별 설명** | (1) **8kHz**: 전화 음성(PSTN) 표준. 대부분의 콜센터 녹취 데이터가 8kHz (2) **16kHz**: 와이드밴드 음성. VoIP, WebRTC 등 디지털 통화에 사용 (3) **48kHz**: 스튜디오/미디어 품질. 콜센터에서는 비일반적 |
| **수치 레퍼런스** | NH투자증권 미팅: TTS 품질 논의에서 *"특히나 48k 되게 고음질 음성이면 모르겠지만 저희 코링 프라이에서 쓰는 거 8k 16k 이 정도면 제 귀로는 (차이 못 느낀다)"* — 실무에서 8kHz/16kHz가 표준이며 48kHz는 예외적 요구 |
| **대응 방안** | "기본 8kHz/16kHz 모두 지원하며, 입력 음성이 48kHz인 경우 자동 다운샘플링 처리합니다. 다만 매뉴얼에 명시적 기재가 없었다면 기술 문서에 추가하여 전달드리겠습니다." |
| **주의사항** | **기술 문서 미비 자체가 검수 불합격 사유**가 될 수 있으므로, 매뉴얼 업데이트 즉시 진행 |

#### Q4. 호전 확률 99%를 달성 가능한가요?

| 항목 | 정답 (참고) |
|------|-----------|
| **핵심 포인트** | **호전(Call Transfer) 확률 99% = 100건 중 99건을 올바른 담당자에게 연결.** 현실적으로 **달성 불가능에 가까운 수치** |
| **99% 불가능 이유** | (1) STT 인식 오류: 별칭/줄임말 오인식 → 잘못된 부서 연결 (2) 발신자 의도 모호성: "아 그 담당자 있잖아, 지난번에..." → AI 특정 불가 (3) 조직 변동: 인사이동/부서 개편 시 매핑 미갱신 (4) 부재/통화중: 정확히 연결해도 담당자 부재 시 실패 처리 여부 |
| **현실적 목표치** | **1단계**: 90~93% (도메인 사전 기반) → **2단계**: 95% (6개월 운영) → **최종**: 97% (1년 이상, 예외 룰 추가) |
| **수치 레퍼런스** | 대신증권 미팅: LG U+ 콜봇이 NPS +50포인트, 대화 건수 60% 감소 효과를 달성했으나, 이는 *"데일리 피드백 루프"*를 통한 지속적 개선과 *"처음에 30명 정도... 지금 3명"* 투입의 결과. 룰 베이스로는 정형 시나리오에서 높은 정확도 가능하나 비정형(별칭/줄임말) 환경에서는 90%+ 달성도 도전적 |
| **대응 방안 (핵심)** | "99%는 기술적으로 매우 도전적인 수치입니다. 현실적으로 1단계 90%, 2단계 95%, 최종 97% 로드맵을 제안드립니다. 나머지 3~5%는 교환원 보조 연결로 커버하는 하이브리드 구조가 가장 안정적입니다. **99%를 약속하고 달성 못 하면 계약 불이행이 되므로, 달성 가능한 목표를 설정하는 것이 양측 모두에게 이로운 접근입니다.**" |

| **NLP vs RAG 비교** | |

| 항목 | NLP | RAG |
|------|-----|-----|
| **정의** | 자연어 처리 기술 전반 (형태소 분석, 의도 분류, 개체명 인식 등) | LLM이 외부 지식DB를 검색(Retrieval)하여 답변을 생성(Generation)하는 방식 |
| **범위** | 넓음 — STT, 텍스트 분류, 감성 분석 포괄 | 좁음 — 지식 기반 Q&A 특화 |
| **관계** | RAG는 NLP 기술의 한 응용 분야 | NLP 기술 위에 구축됨 |
| **RFP 맥락** | "NLP 기반 의도 분류" = 발신자 의도 파악 | "RAG 기반 답변" = 지식DB 검색 후 생성형 AI 답변 |

| 항목 | 정답 (참고) |
|------|-----------|
| **수치 레퍼런스** | 대신증권 미팅: LG U+ 어드바이저가 실제로 RAG 구조를 사용 — *"AWS 몽고DB 아틀라스 기능을 쓰고 있고 LLM 같은 경우에는 GPT랑 제미나이를 하이브리드로 쓰고 있습니다"* (한화손보 미팅). 또한 룰 베이스와 LLM 하이브리드 아키텍처에 대한 논의 — *"두 개를 혼인해서 쓰는데 LLM은 앞에다 넣느냐 뒤에다 넣느냐 여러 가지 아키텍처"* (대신증권 미팅) |
| **대응 화법** | "NLP는 자연어 처리 기술 전체를 아우르는 개념이고, RAG는 그 중에서 '지식 기반 검색+생성' 방식입니다. RFP에서 NLP로 표기된 부분은 의도 분류/발화 인식 요건이고, RAG로 표기된 부분은 지식DB 연동 답변 요건으로 이해하고 제안서를 작성하겠습니다." |

**핵심 함정 대응**:

| 함정 | 대응 전략 |
|------|----------|
| **호전 확률 99% 현실 불가 — 대안 제시 필수** | "99%를 약속하면 계약 불이행이 됩니다. 대신 '단계별 목표 + 미달 시 교환원 보조 연결' 하이브리드 구조를 제안하여, 서비스 품질은 99%에 가깝게 유지하되 AI 단독 처리율 목표는 현실적으로 설정합니다." **근거**: 대신증권 미팅에서도 *"보이스봇은 지금 도입해도 솔직히 기대를 하기는 쉽지는 않네요"* — 영업 담당자도 비현실적 기대를 경계 |
| **인사 정보 체계 연동 보안 한계** | "군 인사 정보는 최고 수준의 보안 등급입니다. AI 시스템에 실명/직위를 직접 탑재하는 것은 보안 심사를 통과하기 어렵습니다. 비식별 코드 매핑 + 별도 인증 게이트웨이 구조로 보안 등급을 충족하는 설계가 필수입니다." |
