# Role Playing Agent 리팩토링 프롬프트

> 출처: 2026.03.31 회의록 (Role Playing Agent 리팩토링 논의, 30분 25초)

---

## 배경 및 현황

Role Playing Agent는 AICC 상담사 교육용 AI 시뮬레이션 시스템이다.
AI가 고객 역할을 수행하고, 상담사(수강생)가 실제처럼 응대 연습을 하는 구조다.

### 현재 아키텍처
- 웹 서버 + 브라우저 기반 접속 (로컬 PC에서 UI만 사용)
- LLM 호출 방식: **CLI 서브프로세스** (현재 문제 원인)
- RAG: **크로마DB** (웹 서버 내부에 로컬 구축)
- 패키지 전체 용량: 약 2GB (FastAPI, 크로마DB 등 라이브러리 포함)
- 현재 LLM: Claude Opus 4.6 (외부 API 호출)

### 현재 문제점

| 문제 | 원인 | 심각도 |
|------|------|--------|
| 응답 레이턴시 10~15초 | CLI 서브프로세스 방식으로 LLM 호출 | 높음 |
| 동접자 접속 시 서버 다운 | 블로킹 방식 → 싱글 스레드 처리 | 높음 |
| 패키지 크기 2GB | 불필요한 라이브러리 포함 | 중간 |
| 시나리오 품질 저하 | 내부 회의록, 잡담 등 노이즈 데이터 포함 | 중간 |
| 고객 AI 응답이 너무 장황 | 프롬프트 미정제, 꼬리 질문 과다 | 중간 |

## 리팩토링 요구사항

### 1. LLM 호출 방식 변경 (최우선)

```
[AS-IS] 웹 서버 → CLI 서브프로세스 실행 → LLM 응답 (10~15초)
[TO-BE] 웹 서버 → LLM API 직접 호출 → 응답 (목표: 1초 이내)
```

- CLI 방식 제거, **REST API 방식**으로 전환
- 우선 무료 API (Gemini Flash 등)로 테스트 → 성능 부족 시 Claude API 유료 전환
- RAG(크로마DB) 검색 결과 상위 3개를 컨텍스트로 포함하여 API 호출

---

### 2. 비동기 처리 개선

- 기존 블로킹 → 비동기(async/await) 구조로 전환 (일부 적용됨, 완전 적용 필요)
- 동접자 10명 기준 설계 (최대 트래픽의 70% 운영 기준 적용)
- 로드 테스트 수행: Apache JMeter 또는 Locust 활용

---

### 3. 시나리오 난이도 개편

```
[기존] 초급 / 중급 / 고급
[변경] 초급 / 중급 / 고급
```

| 레벨 | 방식 | 힌트 |
|------|------|------|
| 초급 | 객관식 (멀티플 초이스) | 없음 |
| 중급 | 서술형 (자유 응답) | 있음 — 답변창 위에 파란색으로 핵심 키워드/포함 내용 안내 |
고급 서술형 (힌트없음) 

**힌트 UI 예시:**
> 💡 다음 키워드를 포함해서 답변해보세요: `STT 인식률`, `콜봇 응답 구조`, `에스컬레이션 조건`

---

### 4. AI 고객 응답 프롬프트 개선

현재 AI 고객이 말이 너무 많고, 꼬리 질문이 과도하게 깊어지는 문제가 있다.

**개선 지침:**
- 응답은 **간결하고 핵심만** 전달 (2~3문장 이내)
- 꼬리 질문은 **실제 고객 발화 데이터 기반**으로만 생성
- 질문 뎁스가 본 시나리오 범위를 벗어나면 원래 주제로 복귀
- 고급 시나리오 삭제로 무한 심화 질문 구조 제거

**AI 고객 역할 프롬프트 수정 방향:**
```
당신은 AICC 고객 역할을 합니다.
- 실제 고객이 발화했던 질문 패턴만 사용하세요.
- 한 번에 하나의 질문만 하세요.
- 답변이 충분하면 다음 주제로 넘어가세요.
- 불필요한 배경 설명, 꼬리 질문, 심화 질문은 하지 마세요.
```

---

### 5. 데이터 품질 관리

- 시나리오 데이터에서 **내부 회의록, 잡담, 내부 논의 내용** 제거
- 파일명 규칙 강제화:
  - 고객 미팅: `YYYYMMDD_고객명_주제.txt`
  - 내부 회의: `YYYYMMDD_내부_주제.txt` (파일명에 "내부" 포함 시 RAG 수집 제외)
- 시나리오 선별 기준: **실제 고객 발화 기반** 질문만 포함

---

### 6. 인프라 방안 (참고)

| 방안 | 구성 | 시점 |
|------|------|------|
| 방안 1 | 현재 서버 유지 + CLI→API 전환 | 즉시 (MVP 개선) |
| 방안 2 | 프라이빗 서버(DGX) + 로컬 LLM | 5월 DGX 구매 후 |

> 현재는 **방안 1** 집중. DGX 도입 전 목업 완성 후 성능 검증.

## 작업 우선순위

```
1순위: CLI → API 방식 변경 (레이턴시 개선)
2순위: 로드 테스트 (동접자 10명 기준 병목 확인)
3순위: 시나리오 데이터 정제 (내부 회의록 제거 + 난이도 재편)
4순위: AI 고객 프롬프트 간결화
5순위: 패키지 경량화 (불필요 라이브러리 제거)
```
